3 Технологическая часть

Технологическая часть выпускной квалификационной работы посвящена описанию программной реализации веб-приложения для цифровизации контрольных листов наблюдения и автоматизации подготовки решения по премированию сотрудников предприятия общественного питания.

Разрабатываемая система переводит процесс ведения контрольных листов наблюдения из бумажного формата в электронный. Приложение позволяет вести учетные записи пользователей, поддерживать актуальный справочник сотрудников, работать с шаблонами КЛН, создавать и заполнять контрольные листы, автоматически рассчитывать итоговые показатели, фиксировать простые электронные подписи, архивировать подписанные документы и формировать отчет по премированию за выбранный месяц.

При программной реализации были учтены следующие требования: авторизация пользователей по табельному номеру и паролю, разграничение прав доступа по ролям, работа со справочником сотрудников, создание и редактирование шаблонов КЛН, оформление контрольных листов наблюдения, автоматический расчет результата, подписание документа двумя сторонами, архивирование КЛН и формирование отчетности по премированию.

3.1 Выбор технологического стека разработки

Для реализации веб-приложения выбран стек, включающий TypeScript, NestJS, Vue.js, Vite, PostgreSQL и TypeORM. Дополнительно используются инструменты для разработки, проверки API и администрирования базы данных.

Выбранный стек позволяет реализовать клиент-серверное приложение, в котором клиентская часть отвечает за интерфейс и взаимодействие с пользователем, а серверная часть обрабатывает запросы, проверяет права доступа, выполняет бизнес-логику, работает с базой данных и возвращает результат в формате JSON.

3.1.1 Выбор языка программирования TypeScript

В качестве основного языка программирования выбран TypeScript. Он является расширением JavaScript и добавляет статическую типизацию, что особенно важно при разработке приложения, в котором одновременно используются разные сущности: пользователи, сотрудники, шаблоны КЛН, критерии, заполненные наблюдения, результаты по критериям, подписи и строки отчета по премированию.

Использование TypeScript позволяет заранее описать структуру объектов, передаваемых между клиентом и сервером. Например, при создании КЛН необходимо корректно передать идентификатор сотрудника, идентификатор шаблона, дату наблюдения, комментарий и массив результатов по критериям. Наличие типизации позволяет выявлять ошибки еще на этапе разработки, а не после запуска приложения.

Дополнительным преимуществом является использование одного языка как на backend, так и на frontend. Это упрощает сопровождение проекта, делает код более единообразным и сокращает вероятность ошибок при изменении структуры данных.

3.1.2 Выбор фреймворка NestJS для серверной части

Для реализации серверной части выбран фреймворк NestJS. Он предназначен для создания серверных приложений на платформе Node.js и хорошо подходит для разработки модульных систем с четким разделением контроллеров, сервисов и слоя доступа к данным.

Серверная часть приложения отвечает за:
- авторизацию пользователей;
- выдачу JWT-токена доступа;
- проверку роли пользователя при обращении к API;
- работу со справочником сотрудников;
- работу с учетными записями пользователей;
- создание и изменение шаблонов КЛН;
- создание, подписание и архивирование КЛН;
- формирование отчета по премированию.

В текущей реализации backend логически разделен на следующие модули:

Модуль	Назначение
auth	Авторизация пользователя и выдача JWT-токена
users	Работа с учетными записями пользователей
employees	Работа со справочником сотрудников
templates	Работа с шаблонами КЛН и критериями
observations	Создание, хранение, подписание и архивирование КЛН
reports	Формирование отчета по премированию

Проверка прав доступа реализована не отдельным модулем ролей, а через перечислимый тип ролей пользователя, декоратор `@Roles` и `RolesGuard`. Такой подход соответствует масштабу проекта и позволяет сохранить модульную структуру без избыточного усложнения.

3.1.3 Выбор Vue.js для клиентской части

Для разработки клиентской части выбран фреймворк Vue.js. Он подходит для создания одностраничных веб-приложений и позволяет строить реактивный пользовательский интерфейс, быстро обновляющийся при изменении состояния данных.

Клиентская часть должна обеспечивать удобную работу пользователя с КЛН в течение смены. Интерфейс должен позволять выполнить вход в систему, выбрать сотрудника, открыть шаблон, заполнить критерии, просмотреть результат, подписать документ и перейти к журналу КЛН или отчету по премированию.

В проекте используется Vue 3 с синтаксисом Composition API и файл `App.vue` в качестве основного пользовательского интерфейса. Для обращения к серверной части используется отдельный модуль `api.ts`, в котором сосредоточены все вызовы REST API. Такой подход упрощает поддержку клиентской части и позволяет централизованно изменять маршруты API или формат запросов.

3.1.4 Выбор Vite для сборки клиентской части

Для запуска и сборки frontend используется Vite. Этот инструмент обеспечивает быстрый старт проекта в режиме разработки, удобную локальную сборку и формирование production-версии приложения.

Использование Vite особенно удобно для Vue-приложения, так как он хорошо интегрируется с TypeScript, поддерживает современный модульный формат и сокращает время ожидания при изменении интерфейса.

3.1.5 Выбор PostgreSQL и TypeORM для работы с данными

Для хранения данных выбрана реляционная система управления базами данных PostgreSQL. Она подходит для данного проекта, так как информация в системе имеет четкую структуру, а между сущностями существуют устойчивые связи: один шаблон содержит несколько критериев, один сотрудник может иметь несколько КЛН, а один КЛН содержит несколько результатов и подписей.

Для взаимодействия серверной части с базой данных используется TypeORM. Этот инструмент позволяет описывать таблицы в виде сущностей и работать с ними через объектную модель, не отказываясь при этом от возможности применять точечные SQL-ориентированные запросы там, где это необходимо.

В текущей реализации через TypeORM описаны сущности пользователей, сотрудников, шаблонов, критериев, наблюдений, результатов по критериям и подписей. Отчет по премированию не хранится отдельной таблицей, а формируется динамически на основе данных таблиц сотрудников и наблюдений.

3.1.6 Выбор инструментов разработки и проверки

Для разработки проекта используется Visual Studio Code. Среда поддерживает работу с TypeScript, Vue.js, Node.js и позволяет удобно работать со структурой проекта, конфигурацией и текстовыми материалами ВКР.

Для администрирования базы данных используется pgAdmin. С его помощью можно просматривать таблицы, проверять сохраненные записи, выполнять SQL-запросы и контролировать структуру базы данных.

Для проверки серверной части используется Postman. Через него можно выполнять вход в систему, проверять маршруты API, тестировать создание сотрудников, оформление КЛН, подписание документов и формирование отчета по премированию.

3.2 Архитектура веб-приложения

Разрабатываемая система имеет клиент-серверную архитектуру. Клиентская часть отвечает за отображение интерфейса и обработку действий пользователя, а серверная часть выполняет бизнес-логику, проверку прав доступа, работу с данными и формирование ответа.

Общая схема взаимодействия выглядит следующим образом:

Пользователь
     ↓
Клиентская часть на Vue.js
     ↓ HTTP/REST
Серверная часть на NestJS
     ↓
TypeORM
     ↓
PostgreSQL

После успешной авторизации сервер выдает пользователю JWT-токен. Далее клиентская часть передает этот токен в заголовке `Authorization` при последующих запросах. Сервер с помощью стратегии `passport-jwt` извлекает пользователя из токена и проверяет, имеет ли он право на выполнение запрашиваемого действия.

Например, при создании КЛН клиент отправляет на сервер идентификатор сотрудника, идентификатор шаблона, дату наблюдения и заполненные результаты по критериям. Серверная часть проверяет корректность данных, связывает КЛН с сотрудником, шаблоном и проверяющим пользователем, рассчитывает итоговые показатели и сохраняет документ в базе данных.

Такое разделение ответственности делает систему удобной для сопровождения: интерфейс, бизнес-правила и слой хранения данных не смешиваются между собой.

3.3 Организация базы данных

База данных используется для централизованного хранения всей информации, необходимой для работы системы. В ней сохраняются учетные записи пользователей, карточки сотрудников, шаблоны КЛН, критерии шаблонов, экземпляры оформленных КЛН, результаты по критериям и данные электронной подписи.

Основные сущности базы данных представлены в таблице 1.

Таблица 1 — Основные сущности базы данных

Сущность	Назначение
users	Хранение учетных записей пользователей системы
employees	Хранение карточек сотрудников предприятия
checklist_templates	Хранение шаблонов контрольных листов наблюдения
checklist_criteria	Хранение критериев, входящих в шаблон

Следует отметить две особенности текущей модели данных.

Во-первых, роли пользователей не вынесены в отдельную таблицу `roles`, а задаются перечислимым типом в поле `role` таблицы `users`. Это соответствует масштабу системы и упрощает хранение ограниченного набора ролей.

Во-вторых, отчет по премированию не хранится в таблице `bonus_reports`. Он формируется динамически в модуле `reports` на основании таблиц `employees` и `observations`. Такой подход исключает дублирование данных и позволяет каждый раз получать отчет по актуальному состоянию системы.

Связи между таблицами организованы по принципу «один ко многим». Один шаблон может содержать несколько критериев, один сотрудник может иметь несколько КЛН, один КЛН включает несколько результатов по критериям и две подписи сторон. Для результатов и подписей настроено каскадное удаление, чтобы при удалении наблюдения не оставались несвязанные записи.

Важной особенностью реализации является ограничение срока хранения КЛН. В системе действует правило хранения наблюдений не более трех месяцев. Очистка устаревших КЛН выполняется автоматически на серверной стороне перед чтением журнала, получением отдельного документа и формированием отчета. Это позволяет соблюдать ограничение хранения без введения дополнительных таблиц архива.

Например, при запросе отчета по премированию клиент передает месяц. Сервер получает список активных сотрудников, отбирает подписанные и архивные КЛН за указанный период, группирует их по сотрудникам, подсчитывает количество наблюдений с нарушениями и формирует итоговое решение по премии.

В приложении реализовано разграничение доступа по ролям. Это необходимо для того, чтобы пользователь видел только те разделы и действия, которые соответствуют его обязанностям.

3.7 Реализация ключевых механизмов системы

3.7.1 Механизм создания и заполнения КЛН

Создание контрольного листа начинается с выбора сотрудника, шаблона и даты наблюдения. После выбора шаблона клиентская часть отображает список критериев, входящих в данный шаблон.

Для каждого критерия пользователь фиксирует результат выполнения, а при необходимости добавляет комментарий. На стороне клиента формируется структура данных, которая затем передается на сервер.

Серверная часть проверяет:
- существует ли выбранный сотрудник;
- существует ли выбранный шаблон и активен ли он;
- относятся ли переданные результаты к критериям выбранного шаблона;
- не превышает ли оценка по критерию допустимое максимальное значение;
- имеет ли текущий пользователь право оформлять КЛН.

После проверки данные сохраняются в таблицах `observations` и `observation_results`.

3.7.2 Механизм расчета результата КЛН

После подготовки результатов по критериям сервер автоматически рассчитывает итоговые показатели контрольного листа:
- сумму набранных баллов;
- сумму максимально возможных баллов;
- процент выполнения;
- количество невыполненных критериев.

Расчет выполняется по формуле:

Итоговый процент = сумма набранных баллов / сумма максимальных баллов × 100%

Если все критерии выполнены полностью, итоговый результат равен 100%. При наличии нарушений процент снижается, а количество невыполненных критериев увеличивается. Автоматический расчет позволяет исключить ручные ошибки и обеспечить единый подход к оценке.

3.7.3 Механизм подписания контрольного листа

После сохранения КЛН документ должен быть подписан двумя сторонами:
- проверяющим;
- сотрудником, по которому проводилось наблюдение.

Подпись проверяющего может поставить только тот пользователь, под чьей учетной записью был создан конкретный КЛН. Для подписи используется табельный номер этого пользователя, хранящийся в системе.

Подпись сотрудника выполняется после ознакомления с результатом наблюдения. Для этого в интерфейсе вводится табельный номер сотрудника. Сервер сверяет введенное значение с табельным номером сотрудника, указанного в КЛН. При успешной проверке формируется запись подписи.

Подпись хранится в таблице `observation_signatures` и включает:
- роль подписанта;
- отображаемое имя подписанта;
- хеш документа;
- подписанный хеш;
- служебный payload;
- дату и время подписания.

После появления двух подписей статус КЛН автоматически изменяется с `draft` на `signed`. Повторное подписание одной и той же стороной запрещено.

3.7.4 Механизм архивирования и хранения КЛН

После подписания КЛН может быть переведен в архив. Архивирование не создает новую запись в другой таблице, а изменяет статус документа на `archived`. Такой подход позволяет хранить всю историю работы с КЛН в одной сущности `observations`.

Перевод документа в архив доступен только директору, менеджеру и отделу кадров. Архивирование применяется только к уже подписанным КЛН.

Дополнительно в системе реализовано ограничение хранения: КЛН хранятся не более трех месяцев. Устаревшие наблюдения удаляются автоматически серверной частью вместе с зависимыми результатами и подписями. Это правило позволяет не накапливать избыточные данные и соответствует требованиям предметной области.

3.7.5 Механизм формирования решения по премированию

Одной из ключевых функций системы является формирование отчета по премированию за выбранный месяц.

Алгоритм работы реализован следующим образом:

1. Пользователь выбирает отчетный месяц.
2. Система получает список активных сотрудников.
3. Из базы выбираются подписанные и архивные КЛН за выбранный период.
4. Для каждого сотрудника определяется количество КЛН и количество наблюдений, в которых есть нарушения.
5. Если по сотруднику за месяц зафиксировано два и более наблюдения с нарушениями, система указывает, что премия не назначается.
6. Если таких наблюдений меньше двух, система указывает, что премия назначается.
7. Клиентская часть отображает итоговый отчет.

Отчет формируется динамически и содержит по каждому сотруднику количество КЛН, количество наблюдений с нарушениями, средний процент выполнения и итоговое решение по премированию.

3.9 Вывод по технологической части

В технологической части были рассмотрены основные средства и подходы, использованные при разработке веб-приложения для цифровизации контрольных листов наблюдения и автоматизации подготовки решения по премированию.

Для реализации серверной части выбран NestJS, для клиентской части — Vue.js и Vite, для хранения данных — PostgreSQL, а для взаимодействия приложения с базой данных — TypeORM. Система построена по клиент-серверной архитектуре, где обмен данными между frontend и backend выполняется через REST API, а доступ пользователей контролируется с помощью JWT-аутентификации и проверки ролей.

В ходе программной реализации были описаны структура базы данных, API-интерфейсы, роли пользователей и основные прикладные механизмы системы: создание и заполнение КЛН, расчет итоговых показателей, простая электронная подпись, архивирование, ограничение срока хранения документов и формирование отчета по премированию.
