СОЗДАНИЕ И НАСТРОЙКА БАЗЫ ДАННЫХ POSTGRESQL ДЛЯ ВЕБ-ПРИЛОЖЕНИЯ КЛН

1. Что важно понимать перед настройкой

В проекте используется PostgreSQL и ORM-библиотека TypeORM.

Если база данных уже была создана ранее, таблицы уже существуют и backend успешно подключается к ней, пересоздавать базу данных и повторно выполнять файл docs/schema.sql не требуется.

Файл docs/schema.sql нужен в двух случаях:
- при полностью новой установке проекта;
- при ручном создании структуры БД без автоматической синхронизации TypeORM.

2. Рекомендуемые параметры базы данных

Рекомендуемое имя базы данных: kln_vit
Рекомендуемый пользователь для локальной разработки: postgres

3. Быстрое создание новой базы данных

Откройте PowerShell и выполните:

psql -U postgres

В консоли PostgreSQL выполните:

CREATE DATABASE kln_vit WITH ENCODING 'UTF8';

Проверить наличие базы можно командой:

\l

Выйти из psql:

\q

4. Настройка подключения backend

В папке backend создайте файл .env на основе .env.example и укажите параметры подключения:

APP_PORT=3000
FRONTEND_URL=http://localhost:5173
DB_HOST=localhost
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=ваш_пароль
DB_NAME=kln_vit
DB_SYNC=true
JWT_SECRET=change_me_for_production
JWT_EXPIRES_IN=8h

Пояснения:

DB_SYNC=true подходит для локальной разработки и демонстрации. В этом режиме TypeORM может автоматически создать недостающие таблицы и поля.

Если база уже существует, DB_SYNC=true также может автоматически добавить недостающие поля, например employee_number, без ручного пересоздания таблиц.

Для более жесткого режима эксплуатации рекомендуется DB_SYNC=false и использование schema.sql или миграций.

5. Ручное создание структуры БД

Если требуется создать структуру вручную, выполните:

psql -U postgres -d kln_vit -f .\docs\schema.sql

После этого можно запускать backend.

6. Основные таблицы

users - учетные записи пользователей системы, их роли, номер сотрудника, табельный номер и раздельно хранимые поля фамилии, имени и отчества.

employees - сотрудники предприятия, по которым оформляются КЛН, с номером сотрудника, табельным номером и раздельно хранимыми полями фамилии, имени и отчества.

checklist_templates - шаблоны контрольных листов наблюдения.

checklist_criteria - критерии внутри шаблонов.

observations - заполненные контрольные листы наблюдения.

observation_results - результаты по каждому критерию.

observation_signatures - электронные подписи проверяющего и сотрудника.

7. Связи таблиц

checklist_criteria.template_id связан с checklist_templates.id.
observations.employee_id связан с employees.id.
observations.observer_id связан с users.id.
observations.template_id связан с checklist_templates.id.
observation_results.observation_id связан с observations.id.
observation_results.criterion_id связан с checklist_criteria.id.
observation_signatures.observation_id связан с observations.id.

Важно:

Поле observer_id является техническим именем связи и обозначает пользователя, который выступил проверяющим и оформил КЛН. В прикладной логике это может быть директор, менеджер или инструктор.

8. Идентификаторы сотрудников

В таблицах users и employees используются два разных номера.

employee_number - номер сотрудника. Это открытый идентификатор, который отображается в интерфейсе и в КЛН.

personnel_number - табельный номер. Это закрытый идентификатор, который используется для входа в систему и подтверждения подписи сотрудника.

last_name, first_name, middle_name - отдельные поля ФИО. Они используются в модели данных вместо единого full_name.

9. Роли пользователей

В таблице users роль хранится в поле role.

Технические значения:
- admin
- manager
- observer
- hr
- employee

Прикладная интерпретация:
- admin - директор
- manager - менеджер
- observer - инструктор
- hr - отдел кадров
- employee - работник ПБО

10. Замечание по структуре schema.sql

В schema.sql может присутствовать поле department в таблице employees. В текущей прикладной логике оно не используется для назначения рабочей зоны сотрудника. Рабочая зона определяется шаблоном КЛН, а не карточкой сотрудника.

Если таблица уже создана с этим полем, удалять его не требуется.

Если база данных была собрана по старой версии модели, где использовались поля full_name, is_required или данные CryptoPro-подписи, рекомендуется пересоздать структуру по актуальному schema.sql либо подготовить отдельную миграцию.

11. Архивирование и премирование

Архивирование реализовано через поле observations.status.
Возможные значения: draft, signed, archived.

Для расчета премирования используется поле observations.violations_count.
Если за месяц по сотруднику зафиксировано два и более КЛН с невыполнением критериев, премия не назначается.

12. Стартовые учетные записи

После запуска backend модуль seed автоматически создает стартовые учетные записи и сотрудников.

13. Проверка подключения

После запуска backend в логах должны появиться сообщения о запуске API и проверке стартовых данных.

Адрес backend API:
http://localhost:3000/api

Адрес frontend:
http://127.0.0.1:5173

14. Если база уже создана по schema.sql

Если вы уже создали базу данных по docs/schema.sql и она используется проектом, дополнительных действий по самому schema.sql выполнять не нужно.

В этом случае достаточно:
1. проверить параметры подключения в backend/.env;
2. запустить backend;
3. запустить frontend;
4. убедиться, что приложение открывается и выполняется вход.
