РЕАЛИЗАЦИЯ ВЕБ-ПРИЛОЖЕНИЯ ВЕДЕНИЯ КОНТРОЛЬНЫХ ЛИСТОВ НАБЛЮДЕНИЯ

1. Назначение системы

Разработанное веб-приложение предназначено для цифровизации процесса заполнения контрольных листов наблюдения на предприятии общественного питания. Система позволяет оформлять КЛН в электронном виде, централизованно хранить результаты наблюдений, подписывать документы участниками процесса и автоматически формировать решение по премированию сотрудников за отчетный месяц.

Основное правило премирования реализовано в соответствии с постановкой задачи: если в течение календарного месяца по сотруднику зафиксировано два и более КЛН с невыполнением критериев, премия за отчетный период не назначается.

2. Используемые технологии

Серверная часть реализована на платформе NestJS.
Для работы с PostgreSQL используется ORM-библиотека TypeORM.
Клиентская часть разработана на Vue.js с использованием Vite.
Взаимодействие клиента и сервера осуществляется через REST API.
Проверка API выполняется через Postman.

В системе используется простая электронная подпись. Подпись формируется на основе табельного номера подписанта и хеша документа, связанного с конкретным КЛН.

4. Идентификация сотрудников

В системе используются два разных идентификатора.

Номер сотрудника - открытый идентификатор, который отображается в интерфейсе и в КЛН. Он используется для визуальной идентификации сотрудника, например: 97 Смирнова Дарья.

Табельный номер - закрытый идентификатор, который знает сам сотрудник. Табельный номер хранится в базе данных и используется:
- для входа пользователя в систему;
- для подтверждения подписи сотрудника при подписании КЛН.

Таким образом, в КЛН отображаются номер сотрудника и ФИ, а подтверждение подписи сотрудника выполняется вводом табельного номера.

5. Роли и права доступа

В прикладной логике предусмотрены следующие роли.

Директор:
может оформлять КЛН, подписывать их как проверяющий, просматривать отчеты, архивировать документы и оценивать как работника ПБО, так и инструктора.

Менеджер:
может оформлять КЛН, подписывать их как проверяющий, просматривать отчеты, архивировать документы и оценивать как работника ПБО, так и инструктора.

Инструктор:
может оформлять КЛН и подписывать их как проверяющий, но только в отношении работника ПБО.

Отдел кадров:
может просматривать сотрудников, КЛН и отчет по премированию, но не оформляет и не подписывает КЛН.

Работник ПБО:
не оформляет КЛН и не оценивает других сотрудников. Работник только подтверждает ознакомление с КЛН своей подписью.

Проверяющим в документе может выступать директор, менеджер или инструктор. В коде и структуре БД для этого используется техническое внутреннее обозначение observer. Оно не означает отдельную прикладную должность, а обозначает пользователя, который выполнил наблюдение и оформил КЛН.

Подпись проверяющего может поставить только тот пользователь, под учетной записью которого был создан конкретный КЛН.

6. Логика заполнения КЛН

При создании КЛН пользователь выбирает:
- сотрудника, за которым ведется наблюдение;
- шаблон КЛН;
- дату наблюдения.

Сотрудник в системе не закрепляется за постоянной рабочей зоной. Работник ПБО может быть обучен нескольким позициям, поэтому рабочая зона определяется не карточкой сотрудника, а выбранным шаблоном КЛН. Именно шаблон задает контекст наблюдения: кухня, прилавок, доставка, напитки, зал и другие рабочие позиции.

После выбора шаблона система отображает критерии контроля. По каждому критерию фиксируется результат выполнения и при необходимости комментарий.

Сотрудником, по которому оформляется КЛН, может быть:
- работник ПБО;
- инструктор.

Директор и менеджер могут оформлять КЛН и на работника ПБО, и на инструктора.
Инструктор может оформлять КЛН только на работника ПБО.

7. Автоматический расчет показателей

После сохранения КЛН сервер автоматически вычисляет:
- сумму набранных баллов;
- максимально возможное количество баллов;
- процент выполнения;
- количество невыполненных критериев.

Итог по листу отображается в виде, например, 9/10 и процента выполнения.

8. Электронная подпись

В системе хранятся две подписи:
- подпись проверяющего;
- подпись сотрудника.

Подпись проверяющего:
документ подписывает директор, менеджер или инструктор, создавший КЛН. Для фиксации подписи используется табельный номер пользователя из его учетной записи и хеш документа.

Подпись сотрудника:
после заполнения КЛН сотрудник знакомится с результатом наблюдения на смене и вводит свой табельный номер на устройстве проверяющего. Если введенный табельный номер совпадает с данными сотрудника в базе, система формирует подпись сотрудника на основе этого табельного номера и хеша документа.

Таким образом, простая электронная подпись реализована как подпись по табельному номеру и хешу документа, а поле signed_by_name сохраняет ФИО подписанта на момент подписания.

9. Архивирование

После завершения работы документ может быть переведен в архив.
Архивный КЛН остается в базе данных и участвует в отчетности, но повторно подписывать его нельзя.

10. Отчет по премированию

За выбранный месяц система:
- группирует КЛН по сотрудникам;
- определяет количество наблюдений с нарушениями;
- рассчитывает средний процент выполнения;
- формирует решение о назначении или неназначении премии.

12. Особенности структуры базы данных

Файл docs/schema.sql остается актуальным как эталонная SQL-структура для новой установки проекта.

Если база данных уже была создана ранее и сервер успешно подключается к ней, повторно выполнять schema.sql не требуется.

Если база данных была создана по более ранней версии модели, где использовались поля full_name, is_required или реквизиты CryptoPro-подписи, для перехода к актуальной структуре потребуется миграция либо повторное создание схемы.

В таблицах и enum-типах могут присутствовать технические имена, например observer и observer_id. Они соответствуют прикладному понятию "проверяющий". Эти имена оставлены без переименования, чтобы не ломать уже созданную базу данных и существующие связи.

Для ускорения выборок по отчетам и карточкам наблюдений в базе данных предусмотрены индексы по полям observations.observation_date, observations.employee_id и observation_results.observation_id.

14. Инструкция по запуску

1. Подготовить базу данных PostgreSQL по инструкции из docs/database_setup.txt.
2. Перейти в папку backend.
3. Создать файл .env на основе .env.example.
4. Установить зависимости командой npm.cmd install.
5. Запустить сервер командой npm.cmd run start:dev.
6. Перейти в папку frontend.
7. Создать файл .env на основе .env.example.
8. Установить зависимости командой npm.cmd install.
9. Запустить клиент командой npm.cmd run dev.
10. Открыть приложение в браузере по адресу http://127.0.0.1:5173.

15. Инструкция по работе с приложением

1. Выполнить вход по табельному номеру и паролю.
2. Открыть раздел создания нового КЛН.
3. Выбрать сотрудника, шаблон КЛН и дату наблюдения.
4. Отметить выполнение каждого критерия.
5. При необходимости указать комментарии.
6. Сохранить КЛН.
7. Открыть сохраненный КЛН в разделе КЛН.
8. Проверить номер сотрудника и ФИ сотрудника, номер сотрудника и ФИ проверяющего, рабочую зону и рассчитанные баллы.
9. Подписать КЛН со стороны проверяющего.
10. Передать устройство сотруднику для ознакомления.
11. Сотруднику ввести свой табельный номер и подписать КЛН.
12. После двух подписей документ получает статус "Подписан".
13. По завершении периода сформировать отчет по премированию.

16. REST API

POST /api/auth/login - вход пользователя по табельному номеру.
GET /api/auth/me - данные текущего пользователя.
GET /api/employees - список сотрудников.
POST /api/employees - создание сотрудника.
GET /api/templates - список шаблонов КЛН.
POST /api/templates - создание шаблона.
GET /api/observations - список КЛН.
POST /api/observations - создание КЛН.
POST /api/observations/:id/sign - подписание КЛН.
PATCH /api/observations/:id/archive - архивирование КЛН.
GET /api/reports/bonus?month=YYYY-MM - отчет по премированию.
