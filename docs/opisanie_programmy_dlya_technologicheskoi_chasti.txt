Описание программы для подготовки технологической части ВКР

1. Назначение программы

Программа представляет собой веб-приложение для цифровизации контрольных листов наблюдения (КЛН) и автоматизации подготовки решения по премированию сотрудников предприятия общественного питания.

Система заменяет бумажное ведение КЛН электронным процессом и позволяет:
- авторизовать пользователей по табельному номеру и паролю;
- вести справочник сотрудников;
- создавать и редактировать шаблоны КЛН;
- оформлять КЛН по сотрудникам;
- автоматически рассчитывать итоговые показатели;
- подписывать КЛН проверяющим и сотрудником;
- архивировать подписанные документы;
- формировать отчет по премированию за выбранный месяц.

2. Что важно не исказить в технологической части

При описании программы нельзя подменять фактическую реализацию шаблонными формулировками. Для данного проекта важно сохранить следующие реальные особенности:

- backend реализован на NestJS;
- frontend реализован на Vue 3 и Vite;
- обмен данными выполняется через REST API;
- база данных построена на PostgreSQL;
- доступ к БД со стороны backend реализован через TypeORM;
- роли пользователей не вынесены в отдельную таблицу `roles`, а задаются полем `role` таблицы `users`;
- отчет по премированию не хранится в отдельной таблице `bonus_reports`, а рассчитывается динамически;
- архив КЛН не вынесен в отдельную таблицу, архивирование реализовано через смену статуса на `archived`;
- КЛН хранятся не более 3 месяцев, после чего удаляются автоматически серверной частью;
- отдельной прикладной роли пользователя `employee` в системе нет, подпись сотрудника выполняется через ввод табельного номера в интерфейсе проверяющего.

3. Технологический стек

Основной стек проекта:
- TypeScript;
- NestJS;
- Vue.js 3;
- Vite;
- PostgreSQL;
- TypeORM.

Дополнительные инструменты:
- Visual Studio Code для разработки;
- Postman для проверки API;
- pgAdmin для просмотра и администрирования базы данных.

4. Архитектура приложения

Приложение построено по клиент-серверной архитектуре.

Схема взаимодействия:

Пользователь
↓
Frontend на Vue.js
↓ HTTP/REST
Backend на NestJS
↓
TypeORM
↓
PostgreSQL

Frontend отвечает за интерфейс, отображение данных и отправку запросов.
Backend отвечает за:
- авторизацию;
- проверку JWT-токена;
- разграничение доступа;
- обработку бизнес-логики;
- расчет результатов КЛН;
- подписи;
- архивирование;
- формирование отчета по премированию.

5. Реальная структура backend-модулей

В технологической части нужно указывать реальные модули проекта:

- `auth` — вход в систему и выдача JWT-токена;
- `users` — учетные записи пользователей;
- `employees` — карточки сотрудников;
- `templates` — шаблоны КЛН и критерии;
- `observations` — создание, хранение, подписание и архивирование КЛН;
- `reports` — отчет по премированию.

Важно:
- отдельного модуля `roles` нет;
- отдельного модуля `signatures` нет, логика подписи входит в модуль `observations`;
- отдельного модуля `bonus` нет, вместо него используется модуль `reports`.

6. Основные сущности базы данных

В проекте используются следующие основные таблицы:

- `users` — учетные записи пользователей;
- `employees` — сотрудники предприятия;
- `checklist_templates` — шаблоны КЛН;
- `checklist_criteria` — критерии внутри шаблонов;
- `observations` — заполненные КЛН;
- `observation_results` — результаты по критериям;
- `observation_signatures` — подписи КЛН.

Что важно отразить в тексте:
- роль хранится в таблице `users` как enum;
- статус КЛН хранится в `observations.status`;
- статусы КЛН: `draft`, `signed`, `archived`;
- в `observation_signatures` хранится роль подписанта, имя подписанта, хеш документа, подписанный хеш и время подписи;
- подписи одной и той же стороны повторяться не могут;
- результаты и подписи связаны с КЛН каскадно.

7. Роли пользователей и права доступа

В системе есть четыре реальные роли:

- `admin` — директор;
- `manager` — менеджер;
- `observer` — инструктор;
- `hr` — отдел кадров.

Распределение прав:

- Директор:
  управление пользователями, сотрудниками и шаблонами; создание КЛН; подпись как проверяющий; просмотр журнала; архивирование; просмотр отчета.

- Менеджер:
  работа с сотрудниками и шаблонами; создание КЛН; подпись как проверяющий; просмотр журнала; архивирование; просмотр отчета.

- Инструктор:
  просмотр списка сотрудников; создание КЛН; подпись как проверяющий; просмотр журнала КЛН.

- Отдел кадров:
  управление учетными записями пользователей; работа с сотрудниками; просмотр журнала КЛН; архивирование; формирование отчета по премированию.

Важно:
- у отдела кадров нет отдельного раздела шаблонов в пользовательском интерфейсе;
- сотрудник, которого оценивают по КЛН, не является отдельным авторизуемым пользователем системы по прикладной логике;
- подпись сотрудника ставится через ввод табельного номера в карточке конкретного КЛН.

8. Основная бизнес-логика

8.1. Создание КЛН

При создании КЛН пользователь выбирает:
- сотрудника;
- шаблон;
- дату наблюдения.

После выбора шаблона загружается список критериев. По каждому критерию фиксируются:
- балл;
- признак выполнения;
- комментарий.

8.2. Расчет результата

После заполнения критериев backend рассчитывает:
- `total_score` — сумма набранных баллов;
- `max_score` — сумма максимально возможных баллов;
- `percentage` — процент выполнения;
- `violations_count` — количество невыполненных критериев.

Формула расчета:

`percentage = total_score / max_score * 100`

8.3. Подписание КЛН

В системе предусмотрены две подписи:
- подпись проверяющего;
- подпись сотрудника.

Логика:
- КЛН сначала находится в статусе `draft`;
- проверяющий подписывает документ под своей учетной записью;
- сотрудник подтверждает ознакомление вводом табельного номера;
- после двух подписей статус меняется на `signed`;
- повторная подпись одной и той же стороны запрещена.

8.4. Архивирование

Архивирование выполняется не переносом в отдельную таблицу, а изменением статуса КЛН на `archived`.

Архивировать можно только уже подписанный КЛН.

Право архивирования есть у:
- директора;
- менеджера;
- отдела кадров.

8.5. Отчет по премированию

Отчет формируется за выбранный месяц.

Логика отчета:
- берутся только КЛН со статусами `signed` и `archived`;
- данные группируются по сотрудникам;
- для каждого сотрудника считается количество наблюдений;
- отдельно считается количество наблюдений, в которых есть нарушения;
- если таких наблюдений два и более, премия не назначается;
- если меньше двух, премия назначается.

Важно:
- отчет не хранится отдельной таблицей;
- отчет рассчитывается каждый раз заново по данным БД.

8.6. Ограничение хранения данных

В системе реализовано правило хранения КЛН не более 3 месяцев.

Это означает:
- устаревшие КЛН удаляются автоматически на backend;
- вместе с ними удаляются связанные результаты и подписи;
- правило очистки применяется до чтения журнала, открытия КЛН и формирования отчета.

9. Реальные API-маршруты, которые нужно упоминать

Если в технологической части приводятся примеры API, нужно использовать реальные маршруты проекта:

- `POST /auth/login` — вход в систему;
- `GET /auth/me` — получение данных текущего пользователя;
- `GET /users` — список пользователей;
- `POST /users` — создание пользователя;
- `PATCH /users/:id` — обновление пользователя;
- `GET /employees` — список сотрудников;
- `POST /employees` — создание сотрудника;
- `PATCH /employees/:id` — обновление сотрудника;
- `GET /templates` — список шаблонов;
- `POST /templates` — создание шаблона;
- `PATCH /templates/:id` — обновление шаблона;
- `GET /observations` — журнал КЛН;
- `GET /observations/:id` — получение одного КЛН;
- `POST /observations` — создание КЛН;
- `POST /observations/:id/sign` — подписание КЛН;
- `PATCH /observations/:id/archive` — архивирование КЛН;
- `GET /reports/bonus?month=YYYY-MM` — отчет по премированию.

10. Основные страницы интерфейса

Во frontend присутствуют следующие основные разделы:

- страница авторизации;
- сводка;
- новый КЛН;
- журнал КЛН;
- премирование;
- персонал или сотрудники, в зависимости от роли;
- шаблоны.

Важно:
- интерфейс зависит от роли пользователя;
- у инструктора в меню вместо общего раздела `Персонал` отображается раздел `Сотрудники`;
- раздел шаблонов доступен не всем ролям;
- архивного раздела как отдельной страницы нет, архивные КЛН находятся в журнале по статусу.

11. Как запустить проект

Для воспроизведения технологической части и проверки программы нужно:

1. Установить Node.js и PostgreSQL.
2. Создать базу данных PostgreSQL.
3. Настроить переменные окружения для backend и frontend.
4. Инициализировать структуру БД по SQL-файлу из `docs/schema.sql` или через подготовленный сценарий настройки.
5. Установить зависимости отдельно для backend и frontend.
6. Запустить backend.
7. Запустить frontend.

Основные команды:

Backend:
- `npm install`
- `npm run build`
- `npm run start:dev`

Frontend:
- `npm install`
- `npm run build`
- `npm run dev`

По умолчанию:
- backend работает на `http://localhost:3000/api`;
- frontend работает через Vite на `http://127.0.0.1:5173`.
